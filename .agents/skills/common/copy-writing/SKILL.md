# Web App Copywriting

## Overview
Copy for the web app must be clear, concise, and user-centric. Avoid marketing fluff, robotic error messages, and passive voice. Good copy helps the user accomplish their task without getting in the way.

## When to Use
- Writing UI labels (buttons, tabs, navigation)
- Creating error messages or success toasts
- Writing empty state messaging
- Drafting onboarding flows or tooltips
- Revising overly formal or "AI-sounding" text

## Core Principles

1. **Be Clear, Not Clever:** Users scan; they don't read. Tell them exactly what is happening or what to do.
2. **Be Concise:** Cut unnecessary words. If a sentence can be shorter, shorten it.
3. **Use Active Voice:** "You updated the ticket" not "The ticket was updated."
4. **Speak Human:** Avoid system jargon like "Authentication failed" or "Invalid parameter." Use "Incorrect password" or "Please enter a valid email."
5. **No Marketing Fluff:** Banish words like "revolutionary," "empower," "unprecedented," "seamless," and "synergy." State the value plainly.

## Quick Reference

| Bad (Without Skill) | Good (With Skill) | Why? |
|---------------------|-------------------|------|
| "The revolutionary repair platform." | "Manage your repair shop." | Removes fluff, states value plainly. |
| "Authentication credentials invalid." | "Incorrect email or password." | Speaks human, avoids system jargon. |
| "Your profile was successfully updated."| "Profile updated." | More concise, active voice. |
| "Click here to proceed to checkout." | "Checkout" | Buttons should be action verbs. |

## Common Mistakes & Red Flags

**Red Flags - STOP and Revise:**
- You wrote "seamless," "empower," or "revolutionize"
- The sentence is longer than two lines
- You used passive voice ("was created by")
- The error message blames the user ("You entered an invalid...")
- The tone sounds like a corporate press release

**Rationalization Table:**

| Excuse | Reality |
|--------|---------|
| "It sounds more professional." | "Professional" doesn't mean "corporate" or "robotic." Clarity is professional. |
| "I need to explain the system state." | Users don't care about the backend state; they care about what it means for them. |
| "Marketing copy needs to be exciting." | Users want to know what the tool does, not how excited we are about it. Value over hype. |

## Implementation Examples

### Empty States
**Bad:** "There are currently no repair tickets available in the database to be displayed."
**Good:** "No repair tickets yet. Create a new ticket to get started."

### Error Messages
**Bad:** "System encountered an unexpected exception while processing the transaction payload."
**Good:** "We couldn't process your payment. Please try again."

### Buttons and Actions
**Bad:** "Submit Repair Request Form"
**Good:** "Create Request"

## Writing Process
1. **Draft:** Write the first version (it will likely be too long).
2. **Cut:** Remove 50% of the words. Delete adverbs and adjectives.
3. **Simplify:** Change complex words to simple ones (e.g., "utilize" -> "use", "facilitate" -> "help").
4. **Verify:** Check against the Red Flags list above. If it fails, start over.
